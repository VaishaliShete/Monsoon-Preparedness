# Household Flood Readiness Card

## The problem

Radha is a ground-floor tenant near a stormwater drain in a low-lying Bengaluru layout (e.g. near Bellandur). She lives with her elderly diabetic mother and two school-age kids, and has no car. Last monsoon, water entered at night — she didn't know what to grab first, her mother's medicines got ruined, and there was no plan for a mother-in-law who can't move fast.

Generic flood-preparedness checklists don't help households like Radha's: they don't know she has a diabetic elder, no vehicle, or that water enters after dark. What she needs is a plan that names *her* actual risks and tells her exactly what to do.

## The solution

Radha (or anyone) types or pastes one messy paragraph describing their household — who lives there, floor level, vulnerabilities, past flooding experience. In seconds, the app returns a single-page, personalized **Family Flood Readiness Card**:

- A prioritized checklist that names the household's actual people and risks
- A concrete evacuation trigger ("if X happens, leave now")
- Medicine and document safety steps
- An emergency contact template to fill in
- The same card rendered bilingually, in English plus Kannada, Hindi, or Tamil — regenerated by the LLM for natural phrasing, not machine-translated word-for-word

Three one-click sample households are included so the flow can be demoed without typing.

## Feature → problem mapping

| Feature | Problem it solves |
|---|---|
| Freeform paragraph input (messy, voice-to-text friendly) | Radha won't fill out a structured form during a crisis or in general; she talks about her life the way she'd tell a neighbour |
| LLM extraction → structured JSON profile | Turns an unstructured story into a profile the app can reliably reason about and validate |
| LLM generation of a prioritized, personalized plan | Generic checklists ignore that her mother is diabetic and can't move fast; this plan is built around her specific situation |
| Explicit evacuation trigger | Last monsoon she "didn't know what to grab first" at night — a concrete trigger removes the moment of indecision |
| Medicine & document safety section | Her mother's insulin was ruined last year; this is called out directly instead of buried in a generic list |
| Emergency contact template | No plan existed for who to call or where to go; the template forces that decision to be made in advance, calmly |
| Bilingual (English + Kannada/Hindi/Tamil) card, regenerated (not MT) | Preparedness plans that aren't in a language the household is comfortable with don't get used in an emergency |
| Graceful fallback UI on malformed LLM output | An emergency-prep tool that silently breaks is worse than none; users see a clear retry path instead of a blank screen |
| Sample household buttons | Lets anyone (a judge, a first responder, an NGO worker) see the personalization instantly without writing a paragraph |

## Google services used, and why

- **Gemini (`gemini-2.5-flash`) via `@google/genai`** — used for three distinct, meaningful jobs:
  1. **Extraction + generation in one structured call.** The messy household paragraph is sent with a `responseSchema` (JSON schema enforced by Gemini's structured output) that describes the full readiness card shape — summary, risk tier, evacuation trigger, checklist, medicine/document steps, contact template. This keeps extraction and generation as a single call for the demo, while still requiring a strict, parseable, typed shape.
  2. **Personalized generation, not a template.** The system prompt explicitly instructs Gemini to name the household's actual people and vulnerabilities inside the checklist reasons and evacuation trigger, so two different households never get the same plan.
  3. **LLM-driven localization.** A second Gemini call takes the generated English card and regenerates it idiomatically in the selected regional language (Kannada, Hindi, or Tamil), rather than piping it through a literal machine-translation API. This was a deliberate choice: Google Translate would produce word-for-word output, which the product spec explicitly calls out as insufficient for a life-safety document that needs to read naturally to a native speaker.

No other Google Cloud service is used — Translate/Maps were considered for Phase 3, but Gemini-based localization was chosen instead so the "not machine-translated word-for-word" requirement and the bilingual demo requirement are met by the same mechanism, without adding a second API and credential to manage.

## Assumptions

- The Gemini API key is a client-side env var (`VITE_GEMINI_API_KEY`), not proxied through a backend. This is acceptable for a local demo/prototype but **is not safe for a public deployment** — anyone with access to the built JS bundle can read the key. A thin Node/Express proxy would be the next step before shipping this publicly.
- One combined Gemini call handles both "extraction" and "generation" for the primary card, rather than two separate round trips, to keep the demo fast and the code simple; the structured JSON schema still gives a distinct, inspectable "profile" shape.
- Risk tier (`moderate` / `high` / `severe`) is inferred by Gemini per household rather than by a separate rules-based classifier.
- Localization only translates the plan's free-text content; the risk badge label and static UI chrome remain in English for this build.
- No backend/database — all state is in-memory in the browser (including a simple in-memory cache of generated/translated cards, cleared on reload).

## How to run

```bash
npm install
cp .env.example .env      # then add your real Gemini API key
npm run dev                # start the dev server
```

Other scripts:

```bash
npm run build     # production build
npm run test        # run the Vitest suite
npm run lint       # oxlint, zero warnings expected
```

Get a Gemini API key from [Google AI Studio](https://aistudio.google.com/apikey) and put it in `.env` as `VITE_GEMINI_API_KEY`. Never commit `.env` — it's gitignored.
